# Sandbox VM P0 Hardening Design

## Goal

Make `@monad/sandbox-vm` a reliable containment backend rather than a boot-and-exec prototype. A cancelled run must stop inside the guest, a failed boot must leave no processes behind, a VM must never survive a policy identity change, and concurrent runs inside a reused VM must not share a process or temporary-filesystem boundary.

The VM remains reusable by agent. Its root disk and mounted workspace persist for the lifetime of that VM; process state, cgroup state, and `/tmp` do not persist between runs.

## Scope

This change covers four P0 areas:

1. A versioned, bounded vsock execution protocol with real stdin, signal, disconnect, and exit semantics.
2. Transactional VM boot plus continuous supervision of the VMM and sidecars.
3. A complete VM identity derived from every policy and artifact dimension that affects containment.
4. Per-run PID, mount, temporary-directory, process-group, and cgroup isolation inside a reused VM.

PTY allocation and terminal emulation remain outside this P0. The protocol reserves a resize message so a later PTY implementation does not require another wire redesign; a resize sent to a non-PTY run returns a structured unsupported response.

## Considered approaches

### Patch the existing one-request protocol

Adding a kill frame and boot cleanup would be the smallest change, but it would leave stdin, orphan reaping, frame bounds, per-run isolation, and future PTY work unresolved. It also keeps lifecycle state implicit in socket ownership.

### Native guest supervisor with protocol v2

The selected design keeps the existing CoreOS image and vsock transport. The vendored Go agent becomes a broker plus a per-run namespace supervisor. It provides the required lifecycle semantics without adding a container runtime to the guest.

### Container runtime inside the VM

Podman or systemd-nspawn would supply mature namespaces and cgroups, but would increase image assumptions, boot cost, attack surface, and operational dependencies. The requested isolation does not require a full image/container model.

## Architecture

### Host transport

`packages/sandbox-vm/src/exec/` owns a protocol-v2 client and process adapter. The first host frame is a bounded JSON `start` request containing:

- protocol version;
- generated run ID;
- argv, cwd, and environment;
- resource limits;
- PTY intent, initially always false.

Subsequent host frames carry stdin bytes, stdin close, signal, or resize. Guest frames carry started metadata, stdout, stderr, structured errors, and a terminal exit record containing exit code and signal.

Every frame has a fixed header and an explicit maximum payload. JSON control frames are limited to 1 MiB and stream frames to 64 KiB. Unknown frame kinds, invalid lengths, duplicate starts, and version mismatches terminate the connection without starting a workload.

`SandboxProcess.kill(signal)` sends a signal frame. Destroying the connection is a cancellation request, not a synthetic exit: the guest supervisor terminates the process group and confirms exit. If the guest cannot confirm within the host grace period, the backend invalidates and tears down the VM.

The process adapter exposes writable stdin and the guest workload PID through the existing `SandboxProcess` seam. Signal names and numeric signals are normalized before crossing the wire.

### Guest broker and run supervisor

The systemd service remains the trusted root broker. It accepts only host-initiated vsock connections and validates the full start request before creating resources.

For each run it starts the same binary in an internal supervisor mode inside new PID and mount namespaces. That supervisor becomes PID 1 for the run, mounts a private tmpfs on `/tmp`, forks the workload as the unprivileged `monad` user, forwards requested signals, and reaps every descendant. The workload has its own process group.

The broker creates a cgroup-v2 child beneath its service cgroup, applies the configured memory and PID ceilings, starts the supervisor in that cgroup, and removes it after the run exits. CoreOS is expected to provide cgroup v2; failure to create the run namespace or cgroup fails closed before the workload starts.

Connection loss triggers `SIGTERM` to the workload process group, followed by `SIGKILL` after a bounded grace period. Agent shutdown performs the same sequence for every active run.

The broker never trusts workload-provided IDs, paths for cgroup creation, or frame sizes. Run IDs are generated by the host and validated as bounded safe identifiers; the broker also prevents duplicates among active runs.

### VM lifecycle transaction

Boot is represented by an internal transaction that records each acquired resource:

- bundle;
- gvproxy;
- virtiofsd processes;
- socat bridge;
- VMM handle;
- live-pool registration.

If any stage fails, rollback stops resources in reverse order and removes the bundle. Driver handles expose all owned sidecars through one idempotent `stop()` operation.

After readiness, the backend observes VMM and gvproxy exit promises. Unexpected exit invalidates the pool entry immediately. Active runs fail with a backend-unavailable error, and the next acquire creates a new VM. VMM and sidecar output is continuously drained into bounded diagnostic tails so pipes cannot deadlock.

Pool teardown remains idempotent and safe when an exit observer and an explicit dispose race.

### Effective policy identity

The VM key is derived from a canonical `EffectiveVmIdentity` containing:

- network mode and filtering proxy endpoint;
- sorted writable and readable mounts;
- sorted read-deny roots;
- sorted masked-file real/fake mappings;
- VM image digest;
- vendored guest-agent digest;
- protocol version;
- ignition/policy schema version;
- resource-isolation settings that shape the guest.

Canonicalization preserves the distinction between undefined, empty, and explicitly unrestricted policy values. Sensitive values are never included; only paths, modes, versions, and content digests enter the identity.

Any identity change creates a distinct VM. Agent disposal still removes all identities belonging to that agent, but correctness no longer depends on the caller remembering to dispose after a policy update.

## Public contracts and configuration

`SandboxSpawnOptions` gains an optional typed per-run limit object. The VM backend supplies safe configured defaults when a caller omits it. The initial settings are:

- maximum processes per run;
- maximum memory per run;
- termination grace period.

These are normal sandbox backend settings persisted in `config.json`; no environment variables are introduced. CPU and wall-clock limits remain owned by the existing tool/process layer unless a later contract explicitly moves them into sandbox policy.

The generic process contract gains a typed stdin handle and structured exit metadata without importing Bun or daemon types into `@monad/sdk-atom`.

## Error handling

- Invalid requests fail before spawn with a structured protocol error.
- Namespace or cgroup setup failure is fail-closed.
- A signal to an exited run is idempotent.
- Abrupt transport loss cancels the real workload.
- Boot rollback errors are collected for diagnostics but do not hide the original boot failure.
- Unexpected VMM/sidecar death makes the pool entry unusable before another run can acquire it.
- A stale or incompatible guest agent fails the version handshake and causes VM teardown.

## Testing

Unit tests cover:

- every protocol frame and malformed/oversized input;
- stdin, signal, disconnect, exit, and duplicate-finish behavior;
- namespace-supervisor process-tree termination using the Go testable seams;
- policy canonicalization and every identity dimension;
- rollback at each boot stage;
- sidecar/VMM unexpected exit invalidation;
- concurrent teardown races and idempotency;
- bounded log tails.

The real-VM conformance suite adds host-side-oracle tests proving:

- cancelling a long-running command stops it in the guest;
- descendants do not survive cancellation;
- one run cannot see or signal another run's processes;
- `/tmp` is private between runs;
- PID and memory ceilings are enforced;
- a failed boot leaves no VMM or sidecar processes;
- changing a deny or artifact identity prevents reuse.

The integration suite remains explicitly gated by `MONAD_VM_IT=1`, but CI should gain at least one KVM runner that executes it. Unit tests must not claim real-VM verification.

## Compatibility and rollout

Host and guest binaries ship from the same package, so protocol v1 compatibility is not retained. Protocol and agent digests in the VM identity prevent a pre-existing bundle from being reused after upgrade.

Existing VM settings keep their defaults. The new per-run limits receive conservative defaults sized below the VM-wide memory and CPU allocation. No migration of durable user data is required because VM bundles are disposable cache state.

## Out of scope

- Windows VM support;
- interactive PTY allocation;
- snapshots and suspend/resume;
- richer filesystem deny carve-outs;
- proxy authentication and dynamic domain approval;
- user-selectable guest distributions.

Those remain follow-up work after the four P0 invariants are verified.
