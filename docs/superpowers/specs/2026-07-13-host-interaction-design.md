# Host Interaction Design

## Goal

Provide a transport-neutral way for third-party atoms to request bounded user input through Web, TUI, interactive CLI, non-interactive CLI, and capable ACP clients. Atoms declare semantic content with a safe schema; the daemon owns routing, lifecycle, security, and exactly-once completion.

## Contract

Daemon-side atom code calls `ctx.requestInteraction(request)` and awaits an `InteractionResult`. No callback or executable UI crosses a transport boundary.

Supported requests are `confirm`, `select`, and `form`. Form fields are limited to `string`, `secret`, `number`, `boolean`, and `select`, with declarative required, min/max, pattern, default, description, and options metadata. HTML, scripts, frontend components, executable validators, files, and rich-text controls are rejected.

Results are either submitted values or cancellation with `close`, `escape`, `timeout`, `disconnect`, or `unavailable`. Completion is exactly once.

## Request lifecycle

The daemon assigns an interaction ID, records trusted source attribution, and publishes a redacted request. A request initiated by a user action prefers that same client. Background requests enter a shared pending queue and never steal focus.

A presenter claims a request with a short lease. Other clients show that it is being handled elsewhere. Submit and cancel require the interaction ID plus lease token. Disconnect or lease expiry makes a non-secret request claimable again; secret field values are never retained during a failed claim.

Requests have bounded lifetime, per-source concurrency limits, schema size limits, and text length limits. When no compatible presenter exists, foreground requests fail with `interaction_unavailable`; background requests remain pending until timeout.

## Presenter capability negotiation

Each client advertises supported interaction and field types plus secret-input and background-queue capabilities. A presenter may claim only a request it can render completely.

- Web renders host-owned dialogs or sheets.
- TUI renders host-owned modals with keyboard navigation.
- Interactive CLI prompts field-by-field; secret input disables echo.
- Non-interactive CLI returns a structured `interaction_required` result with a resumable interaction ID. `monad interaction answer <id>` opens an interactive presenter later.
- ACP maps supported requests to elicitation. Unsupported schemas remain pending or fail explicitly.

## Security

Every presenter shows a non-spoofable source header generated by the host. Third parties control semantic labels and descriptions but not host chrome, colors, sizing, or attribution.

Secret values never enter events, pending lists, transcripts, logs, shell history, structured CLI output, or retry state. Presenters without safe secret input cannot claim secret-bearing requests. Only the requesting atom receives the submitted result.

## Sandbox integration

Sandbox backend settings use the same schema types. `/studio/sandbox` retains host-owned backend cards and status. Configure or Activate invokes Host Interaction when required fields are missing or need replacement. The Studio implementation contains no contributed-backend conditionals.

## Testing

Tests cover schema validation, redaction, capability matching, preferred-client routing, background queuing, claim leases, exactly-once completion, timeout/disconnect behavior, secret non-retention, Web/TUI/CLI rendering, non-interactive resume, and ACP elicitation mapping.

## Scope boundaries

The first version excludes arbitrary frontend code, file upload, rich text, nested layout schemas, background focus stealing, and durable secret-value drafts.
