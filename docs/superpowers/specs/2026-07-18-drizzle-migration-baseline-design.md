# Drizzle Migration Baseline Design

## Goal

Flatten the unpublished product's SQLite migration history into two intentional migrations: one schema migration generated by Drizzle Kit and one handwritten migration for SQLite FTS5 objects that Drizzle schema definitions do not own.

## Migration Layout

`0000_initial-schema.sql` is regenerated from the current `apps/monad/src/store/db/schema.ts`. It represents the complete current relational schema directly, without historical create-then-drop operations or columns removed by later migrations.

`0001_message-fts.sql` remains handwritten. It creates the two FTS5 virtual tables, their insert/delete/update synchronization triggers, and the initial rebuild statements.

No other SQL migrations or snapshots remain. `meta/_journal.json` contains exactly the generated baseline entry followed by the handwritten FTS entry. The generated migration bundle is rebuilt from this two-entry journal.

## Generation Procedure

Generate a fresh baseline in an isolated temporary output directory with the repository-pinned Drizzle Kit and the current schema. Copy the generated SQL and snapshot into the canonical migration directory, rename the migration tag to `0000_initial-schema`, and preserve the generated metadata content.

Retain the existing handwritten FTS SQL as `0001_message-fts.sql`. Its snapshot represents no additional Drizzle-managed relational schema change, so the metadata must remain consistent with the baseline schema while the journal records the custom migration.

Delete the superseded observation-event creation/removal and `output_snapshot` removal migrations and their snapshots.

## Compatibility

This is intentionally a pre-release history rewrite. Existing local databases that recorded the previous migration hashes are not a compatibility target. New databases must initialize directly to the current schema plus FTS objects.

## Verification

- `bun run db:check` accepts the journal and snapshots.
- `bun run db:bundle` produces an up-to-date `migrations.generated.ts` containing exactly two migrations.
- `bun run db:drift` reports no schema or artifact drift.
- Applying the bundled migrations to an empty SQLite database succeeds.
- The initialized database contains the current relational tables, both FTS virtual tables, and all three FTS synchronization triggers.
- Relevant migration-history, drift, and store tests pass.
