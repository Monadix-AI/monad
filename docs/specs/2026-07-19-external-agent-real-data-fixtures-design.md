# External Agent Real-Data Fixtures Design

## Goal

Capture representative Codex and Claude Code raw observation records from the local Monad instance, remove all user semantics and machine identity, and use the sanitized records as deterministic projector and cursor regression fixtures.

## Source and scope

The one-shot capture uses Monad's public external-agent observation APIs instead of reading provider files directly. It selects usable Codex and Claude Code sessions from the daemon-wide session summaries and reads bounded raw history pages. The capture script is deleted after fixture generation; fixture creation is not production tooling.

The committed fixture is evidence of provider-native shapes accepted by Monad. It is not a replay of a user conversation and must not contain prompts, replies, tool arguments or results, paths, account data, tokens, real session identifiers, or other machine-local values.

## Fixture contract

Each provider has one JSON fixture containing:

- `provider: "codex" | "claude-code"`
- `page`: protocol-valid history coverage and 24 ordered raw records

Provider structural vocabulary required by a projector is preserved, including record `type`, `subtype`, `method`, `event`, `role`, and status-like enum values. All other strings are classified and replaced with deterministic placeholders. Equal input identifiers map to equal placeholders; distinct identifiers remain distinct. Timestamps are normalized while preserving order. Numbers and booleans remain intact so usage, sequence, and state-machine behavior remain testable.

## Sanitization boundary

Unredacted capture data stays in process memory. The capture command sanitizes before calling `Bun.write`; there is no raw intermediate file.

The committed test walks every string and rejects anything outside provider structural vocabulary, deterministic placeholders, and normalized timestamps. The capture also rejects home-directory paths, email addresses, UUIDs, bearer/API-token forms, and the selected sessions' local identifiers before writing.

## Tests

Fixture-driven tests cover:

1. strict fixture and protocol parsing;
2. exact provider projection for representative Codex and Claude records;
3. Codex reasoning and custom tool call/result boundaries;
4. Claude user, thinking, tool call/result, assistant, and system boundaries;
5. repository safety scanning of every non-structural string.

The expected projection contract is authored in tests after inspecting the sanitized fixture. It is not generated by the projector under test.

## Non-goals

- No changes to provider acquisition, observation cursor production, or convenience projection implementation.
- No CI dependency on a running local daemon.
- No committed user content or unredacted capture artifact.
- No exhaustive archive of local Codex or Claude history.
- No reusable fixture generator or live-stream capture.
